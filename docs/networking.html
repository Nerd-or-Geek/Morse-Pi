Networking

Connect multiple Morse-Pi devices on the same network for real-time radio-style communication.

Overview

The NET tab lets Morse-Pi devices on the same LAN discover each other automatically and exchange Morse code in two ways:

  • Store & Forward — key or type a complete message, then send it to a peer. The receiving Pi plays it through its speaker and stores it in the inbox.
  • Live Radio Transmission — select a peer, then every paddle press or key-down is forwarded in real time. The remote Pi's speaker plays the tone as you press — just like a radio contact.

No configuration is needed. Devices discover each other automatically as long as they are on the same subnet.

Peer Discovery

Morse-Pi uses UDP broadcast beacons for zero-configuration discovery:

  1. Every device sends a UDP broadcast to port 5001 every 3 seconds.
  2. The beacon payload is a JSON object: {"name":"<device-name>","ip":"<ip>","port":5000}
  3. Receiving devices add the peer to their peer list and update its last-seen timestamp.
  4. Peers that have not sent a beacon for 15 seconds are automatically removed from the list.
ℹ Broadcast scope
UDP broadcasts are limited to the local subnet. Devices on different VLANs or subnets will not see each other. All Pis must be on the same network segment (e.g., all connected to the same router or access point).
ℹ Firewall
Make sure port 5000 (HTTP) and port 5001 (UDP) are not blocked by a firewall on any of the devices. On Raspberry Pi OS the default firewall is open, so this is usually not an issue.

Setup

  1. Install Morse-Pi on each Raspberry Pi (see Getting Started → Installation).
  2. Connect all Pis to the same Wi-Fi or wired network.
  3. Open the web UI on each Pi (http://<pi-ip>:5000).
  4. Open the NET tab on each Pi.
  5. Set a unique Device Name for each Pi (top of the NET tab). Examples: Shack-1, Portable.
  6. Within a few seconds, each Pi should appear in the other's peer list.
✓ No configuration files to edit
All networking settings are configured live from the NET tab in the web UI. Changes take effect immediately.

Store & Forward Messages

In store-and-forward mode, you compose a complete Morse message on your device and then send it. The receiving Pi plays the message and stores it in its inbox.

Send a typed message

  1. In the NET tab, select a peer from the Peers list.
  2. Choose Message Mode.
  3. Type your message in the text field and click SEND.

Send keyed Morse

  1. Select a peer and choose Key Mode.
  2. Key your message using the physical paddle, on-screen buttons, or keyboard shortcuts (Space / Z / X).
  3. Your keyed dots and dashes are decoded in the buffer display.
  4. Click SEND (or press Enter) to dispatch the decoded text to the peer.
ℹ How it works under the hood
The message is sent via HTTP POST to the peer's /receive_morse endpoint. The receiving Pi plays the message through its speaker using the same Morse timing engine used for local playback.

Live Radio Transmission

Live transmit mode is the real-time radio contact feature. Every key-down/key-up event is forwarded to the selected peer immediately — the remote Pi's speaker tone follows your paddle in real time.

Enabling live transmit

  1. Select a peer from the Peers list in the NET tab.
  2. Toggle the Live Transmit switch to ON.
  3. The NET display shows "hold key or paddle to transmit…" when idle.

Transmitting

Once enabled, use any of these input methods to transmit:

Input methodAction
Physical straight keyPress/release → tone on/off at peer in real time
Physical iambic paddleDot/dash paddle → keyer sends timed elements to peer
On-screen DOT / DASH buttonsHold to transmit that element to peer
Space keyStraight key emulation — hold to transmit
Z keyDot paddle in iambic/dual-pin mode
X keyDash paddle in iambic/dual-pin mode

What the receiving Pi does

  • When a key-down event arrives, the receiving Pi immediately starts playing a tone through its speaker.
  • When the key-up event arrives, the tone stops.
  • The button active indicator on the receiving Pi's UI lights up while the key is held.
  • No message is stored — live transmit is ephemeral, just like a radio contact.
⚠ Latency
Live transmit adds network latency. On a local LAN this is typically <5 ms and imperceptible. Over Wi-Fi with congestion it can reach 20–50 ms. Wired Ethernet gives the most consistent results. Live transmit is not recommended over the internet.
ℹ How it works under the hood
Key-down and key-up events are sent via HTTP POST to the peer's /net_receive_live_key endpoint. The timeout is 600 ms per request with a single attempt to minimize latency. If a packet is lost the tone will be cut short or held slightly longer — the system self-corrects on the next event.

Network Key Modes

The NET tab has three selectable key modes, controlled by the buttons at the top of the NET panel:

ModeDescription
MessageType a text message and send it to a peer.
Key ModeKey Morse on your paddle; the decoded text is buffered and sent when you click SEND.
Live TransmitEvery key event is forwarded in real time — no buffering, no decoding.

Inbox

Received messages (from store-and-forward sends) are stored in the Inbox at the bottom of the NET tab. Each entry shows the sender name, timestamp, and message text.

  • Click a message to replay it through the speaker.
  • Messages persist until you click Clear.
  • Inbox capacity: the 20 most recent messages are kept.
ℹ Live transmit & inbox
Live transmit events do not appear in the inbox. Only store-and-forward messages sent to /receive_morse are logged.

Troubleshooting

No peers appear

  • Confirm both Pis are on the same subnet (e.g., both on 192.168.1.x with the same router).
  • Check that UDP port 5001 is not blocked. Test with: nc -u -l 5001 on one Pi and echo hello > /dev/udp/<other-ip>/5001 on the other.
  • Ensure both Pis are running Morse-Pi and that the service is active: sudo systemctl status morse-pi.
  • Wait up to 5 seconds after starting — the first beacon fires within 3 seconds.

Peer appears but messages fail to send

  • Confirm port 5000 (TCP/HTTP) is reachable: curl http://<peer-ip>:5000/status should return JSON.
  • Check for firewall rules: sudo iptables -L.
  • If the sending Pi shows a timeout error, verify the peer IP hasn't changed (reassign a static IP or use DHCP reservations).

Live transmit tone is choppy or delayed

  • Use wired Ethernet instead of Wi-Fi for the most consistent latency.
  • Reduce other network traffic during the contact.
  • Make sure the pigpiod daemon is running on the receiving Pi: sudo systemctl status pigpiod.
  • If the keyer fires but no tone plays at the receiver, restart the Morse-Pi service: sudo systemctl restart morse-pi.

Live transmit is enabled but nothing happens when I key

  • Make sure a peer is selected in the Peers list before enabling Live Transmit.
  • After selecting the peer, toggle the Live Transmit switch off and back on.
  • Check that the peer is still online (the peer's status indicator should show it as active).
